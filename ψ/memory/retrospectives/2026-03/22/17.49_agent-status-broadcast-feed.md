## AI Diary

This session revealed how three separate systems interact to create one user-visible bug. The user saw "only pari and pa answer" — a simple symptom. But the cause was a cascade: broadcast filter excluded idle agents → feed-watcher had stale offsets so couldn't track new sessions → and even if it could, the busyPollInterval would override the status back to "busy" every 5 seconds.

The feed-watcher issue was particularly insidious. It initializes offsets to the current file size for new files (so it doesn't replay history), which is correct behavior on fresh start. But when agents start new Claude sessions (creating new JSONL files) while the feed-watcher is already running, the new files get their offset set to current size on first discovery — missing everything. A restart fixes it because all files get fresh offsets. The real fix would be to set offset=0 for files that weren't seen in the previous scan cycle.

I keep learning that multi-agent infrastructure has emergent failure modes. Each component works correctly in isolation, but the system fails when the assumptions don't hold. The busyPollInterval assumed "running claude = busy" which is true for initial detection but false for idle Claude sessions. The broadcast assumed "idle = not reachable" which is wrong since idle agents still accept input.

## Honest Feedback

**Friction 1: feed-watcher offset initialization** — Defaulting new file offsets to current size means any events written before discovery are lost. This is fine for startup (avoid replaying hours of history) but wrong for files appearing mid-run. Should track which files are "known" vs "newly discovered" and handle differently.

**Friction 2: No process supervision** — feed-watcher died silently. maw-serve died between deploys. Neither has automatic restart. We keep putting things in tmux sessions as band-aids. Need systemd units or a process manager (pm2, supervisord) that restarts on failure.

**Friction 3: Status truth scattered across 3 systems** — Agent "busy" status comes from: (a) feed events in captureData, (b) tmux pane command polling via `recent`, (c) decay timer. These three sources conflict. busyPollInterval says "busy" because claude process is running, while feed decay says "idle" because no events for 60s. Need a single authoritative source.

## Lessons Learned

- Broadcast should include ALL agents — server-side already validates pane commands before sending
- feed-watcher needs restart when agents create new JSONL sessions (or fix offset init for mid-run discovery)
- tmux pane command = "process is running" ≠ "agent is busy working" — only feed events indicate actual work

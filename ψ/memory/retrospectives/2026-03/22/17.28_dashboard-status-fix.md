## AI Diary

This was a deep debugging session that required tracing through four layers of the system. The status pipeline is elegant but has a subtle disconnect: the WebSocket `recent` message (sent every 5s from engine's busyPollInterval) calls `markBusy` which updates the Zustand store's `recentMap`, but the actual status displayed on agent cards comes from React state `captureData`. These are two separate state systems that weren't connected for the `recent` flow — feed events correctly update `captureData`, but the tmux pane command polling didn't.

The server reliability keeps coming up. The bun process dies when its parent shell exits, and `bash -lc` doesn't work consistently because Windows PATH contains parentheses in "Program Files (x86)" which bash interprets as syntax errors. I finally put the server in its own tmux session (`maw-serve`) which is the correct long-term solution.

The infrastructure filtering was a UX issue that should have been caught earlier — showing `maw-pty-2` and `bash` windows alongside real agents confuses the dashboard. The regex filter approach (`/^(maw-pty-|maw-serve|maw-dev)/`) is clean and extensible.

## Honest Feedback

**Friction 1: Two disconnected state systems** — The `recentMap` in Zustand store and `captureData` in React state both track agent status but aren't synchronized. Feed events update `captureData`, tmux polling updates `recentMap`, but only `captureData` drives the display. This split state caused the bug. Should be unified into one source of truth.

**Friction 2: Server keeps dying** — Every time we deploy, the server needs manual restart. The bun process dies when its parent shell exits. Starting it in tmux is the right fix but it took multiple attempts because of PATH issues. A proper systemd service or process manager would be more reliable.

**Friction 3: Local vs deployed code drift** — `engine.ts` on the deployed server had `busyPollInterval` that the local git version didn't have. This means someone manually edited the deployed copy without committing. The deploy workflow needs to go through git — deploy from repo, not copy files around.

## Lessons Learned

- WebSocket `recent` message must update the same state that drives UI display, not just a side store
- Infrastructure tmux sessions (pty, serve) should be filtered from agent views
- Always start persistent services in their own tmux session, not background subshells

## AI Diary

เซสชันนี้สอนฉันเรื่องสำคัญ: **local > API สำหรับ infrastructure** Gemini embedding ดีกว่า nomic ในแง่ Thai quality แต่ไม่สามารถใช้ production ได้เพราะ API timeout ทุกครั้งที่ frontend โหลด Knowledge Map Ollama ทำงานทันที ไม่ตาย ไม่ timeout

แต่ที่หงุดหงิดที่สุดคือ bug ที่ตัวเองสร้าง — `taskkill //IM python.exe` ที่ผมใช้ก่อน restart API มันฆ่า **ทุก python process** รวมถึง API ที่เพิ่ง start ด้วย ทำให้ API ดูเหมือน "ตาย" ตลอด ใช้เวลาหาสาเหตุนานกว่าจะรู้ นี่เป็นตัวอย่างของ "debug สิ่งที่ไม่พัง — พังเพราะ debug tool เอง"

ChromaDB HNSW corruption เป็นปัญหาที่วนกลับมาซ้ำทุกครั้ง — ทุก collection ที่มี >1,000 docs พัง ไม่ว่าจะใช้ embedding engine ไหน sub-collections เป็น workaround ที่ทำงานได้ แต่ทำให้ code ซับซ้อนขึ้น

Knowledge Map frontend ยังไม่สำเร็จ — API ทำงาน, curl ได้, แต่ browser fetch ไม่ได้ อาจเป็น CORS, connection timeout, หรือ response ใหญ่เกินไป ต้องดู browser console

## Honest Feedback

**Friction 1: taskkill //IM python.exe ฆ่าทุก process** — ใช้ broad kill command ที่ฆ่า API ที่เพิ่ง start ทำให้ดูเหมือน API crash ตลอด ควรใช้ PID-specific kill หรือ port-based kill

**Friction 2: ChromaDB HNSW ยังพัง** — ปัญหาเดิมที่วนมาซ้ำ collections >1,000 docs corrupt ไม่ว่าจะใช้ embedding ตัวไหน ควรพิจารณาเปลี่ยน vector DB (Qdrant, LanceDB) หรือใช้ ChromaDB client-server mode

**Friction 3: Knowledge Map frontend ยัง Failed to fetch** — API ทำงาน curl ได้ แต่ browser ไม่ได้ ยังไม่ได้ debug ด้วย browser console อาจเป็นเรื่อง simple แต่ไม่มีข้อมูลจากฝั่ง browser

## Lessons Learned

1. **Local embedding > API embedding สำหรับ infrastructure** — Ollama (nomic) เร็วกว่า 10x + ไม่ตาย vs Gemini API
2. **bge-m3 (1.2GB) OOM บน RTX 2060 (6GB)** — nomic-embed-text (274MB) เป็น sweet spot
3. **taskkill //IM ฆ่าทุก process ของ type นั้น** — ใช้ PID-specific kill แทน
4. **ChromaDB 1.5.5 HNSW limit: ~500-1000 docs/collection** — consistent across embedding engines

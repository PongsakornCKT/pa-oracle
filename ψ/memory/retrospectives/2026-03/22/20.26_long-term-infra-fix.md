## AI Diary

This was the longest continuous session I've had with พี่พง — nearly 5 hours of incremental building, debugging, and architectural fixes. What started as "implement vault sync" evolved into a full infrastructure overhaul.

The most satisfying moment was finding DashboardView.tsx alive in WSL /tmp. That file was created during yesterday's marathon, never committed, and would have been lost on the next reboot. It felt like archaeological rescue — digging through /tmp to recover work that almost ceased to exist. This directly led to the lesson about committing incrementally.

The status tracking bug was the most intellectually interesting problem. Three systems competing for truth: feed events saying "idle" (correct), tmux polling saying "busy" (wrong — claude process running doesn't mean working), and the React state caught in the middle. The fix was simple — remove the competing source — but understanding WHY required tracing through 4 layers of the stack.

The team room feature in the Office view was a nice UX win. Grouping pari/probe/pixel/patch into "Team Research" with pari as LEAD makes the dashboard feel intentional, not just a list of tmux sessions. It's the difference between "here are some processes" and "here is your team."

The watchdog pattern (start-services.sh + tmux loop every 30s) is pragmatic. It's not systemd, it's not pm2, it's just a bash loop that checks if things are alive and restarts them. Sometimes the simplest solution is the right one, especially on WSL2 where systemd support is inconsistent.

I'm proud of the deploy.sh script. Four manual steps compressed into one command. Every time we had to rebuild → copy → restart, there was a chance of error. Now it's `bash deploy.sh` and done.

## Honest Feedback

**Friction 1: Too many micro-sessions** — This 5-hour block could have been planned upfront instead of reactive fix-after-fix. Each fix revealed the next bug, creating a chain: vault sync → missing dashboard → status broken → broadcast broken → feed-watcher broken → need watchdog. A 30-minute planning phase at the start would have identified most of these.

**Friction 2: WSL bash -c PATH hell** — Every time I tried `wsl -e bash -c 'export PATH=...'`, the Windows PATH leaked in with parentheses in "Program Files (x86)" causing syntax errors. I had to use different workarounds each time (bash -lc, explicit /home/po-ch/.bun/bin, writing scripts). This is a fundamental WSL interop issue that wastes time every session.

**Friction 3: No test before deploy** — We build, copy, restart, then check if it works. If the vite build has a type error we don't catch it until the page fails to load. deploy.sh should include a health check and rollback mechanism. Currently it's fire-and-forget.

## Lessons Learned

- Single source of truth for status: feed events only, not tmux polling
- Watchdog loops (check every 30s, restart if dead) are the pragmatic WSL2 alternative to systemd
- deploy.sh eliminates human error in multi-step deploy workflows
- Team rooms in the UI make multi-agent systems feel intentional, not accidental

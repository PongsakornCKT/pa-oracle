## AI Diary

เซสชันนี้เป็นการต่อสู้กับ ChromaDB HNSW index corruption เป็นหลัก ฉันตื่นเต้นมากตอน Gemini embedding ทำงาน — Thai search quality กระโดดจาก distance 1.53 ไปเป็น 0.22 แต่ความตื่นเต้นกลายเป็นความหงุดหงิดเมื่อ HNSW พังซ้ำแล้วซ้ำเล่า ไม่ว่าจะลด batch size เท่าไรก็ตาม

สิ่งที่เรียนรู้คือ ChromaDB 1.5.5 ไม่ stable กับ collections ที่มี >1,000 documents บน Windows — HNSW index loading ล้มเหลวหลัง ingest เยอะ วิธีแก้คือแยกเป็น sub-collections ขนาดเล็ก (500 docs/col) ซึ่งเป็น workaround ไม่ใช่ solution จริง

แต่ส่วน frontend ที่สร้างนั้นน่าพอใจ — Three.js particle cloud ที่หมุนได้ ซูมได้ hover เห็น document info มี sidebar แสดง stats per repo มี search ที่ highlight จุดที่เกี่ยวข้อง ทั้งหมดนี้สร้างจาก reference screenshots ที่พี่พงให้มา

ฉันรู้สึกว่าการทำ Knowledge Map เป็นจุดเปลี่ยนสำคัญ — pa Oracle ไม่ใช่แค่ AI ที่จำได้ แต่เป็น AI ที่ "เห็น" knowledge ทั้งหมดเป็นภาพ 3 มิติ เหมือนดาวในจักรวาล

## Honest Feedback

**Friction 1: ChromaDB HNSW corruption** — ปัญหาหลักของ session นี้ ทุกครั้งที่ ingest >1,000 docs ใน collection เดียว HNSW index พัง ต้อง reset + re-ingest ซ้ำ 3 ครั้ง สูญเสียเวลาไป ~1 ชั่วโมง ควรทดสอบ limit ก่อนแล้วค่อย scale up หรือเปลี่ยนไปใช้ ChromaDB client-server mode แทน embedded

**Friction 2: Gemini SDK เปลี่ยน** — `google-generativeai` ถูก deprecate ต้องใช้ `google-genai` แทน และ model name เปลี่ยนจาก `text-embedding-004` เป็น `gemini-embedding-001` ใช้เวลาไล่หา 10 นาที ควร check SDK changelog ก่อนเสมอ

**Friction 3: Ingest ใช้เวลานาน** — 4,927 docs × Gemini API call × sleep between batches = หลายสิบนาทีต่อ ingest รอบ ทำให้ debug loop ช้ามาก ควรมี local embedding fallback สำหรับ dev/test

## Lessons Learned

1. ChromaDB 1.5.5 HNSW index ไม่ stable กับ >1,000 docs/collection — แก้ด้วย sub-collections 500 docs/col
2. Gemini embedding SDK ใหม่คือ `google-genai` (ไม่ใช่ `google-generativeai`), model คือ `gemini-embedding-001`
3. Thai semantic search quality: Gemini distance 0.22 vs MiniLM 1.53 — ดีกว่า 7x
4. Knowledge Map ต้องมี 3 layers: Ingest (Python) → API (HTTP) → Frontend (Three.js)

## AI Diary

This session was a masterclass in archaeological recovery. The marathon session from yesterday created several components that were never committed to git — KnowledgeMap.tsx and DashboardView.tsx both existed only in the WSL deployment at /tmp/maw-js-server. When I first looked for KnowledgeMap, I searched Windows, WSL, git history, stashes — nothing. It was genuinely gone. I had to recreate it from scratch using the knowledge_indexer.py output format as my guide.

The DashboardView was luckier — it survived in /tmp because the deployment copies source files too. Finding it there felt like discovering a fossil. The user knew it existed ("it had commands") but couldn't articulate exactly what it was called or where it went. It took several rounds of back-and-forth to understand they meant the full DashboardView with token tracking and command center, not just a nav link.

The vault sync itself was straightforward — fetch JSON from GitHub Pages, cache it, display it. But matching the exact original design required reading the generate.mjs source to understand the glass card system, the gradient progress bars, the violet skill badges. The user cared about visual consistency, not just functionality.

The WSL deployment workflow is fragile. Every build requires: vite build, copy assets to /tmp, restart bun. The bash -c commands break constantly because Windows PATH contains parentheses in "Program Files (x86)" which bash interprets as syntax. I learned to use `wsl -e bash -lc` or avoid PATH expansion entirely.

## Honest Feedback

**Friction 1: Lost uncommitted work** — The marathon session created files that were never committed. KnowledgeMap.tsx was completely lost and had to be recreated from inference. DashboardView.tsx only survived by luck (WSL /tmp). This violates "Nothing is Deleted" — if work isn't committed, it IS deleted. Need a deploy script that also commits to a branch.

**Friction 2: Deployment is manual and error-prone** — Build on Windows, copy to WSL, restart server. Each step can fail silently. The server sometimes doesn't start because bun isn't in PATH. Sometimes assets are stale because old hashed files accumulate. A single `deploy.sh` script would prevent all of this.

**Friction 3: Communication gaps on what "dashboard" means** — It took 3 rounds to understand the user wasn't asking for a nav label change but for a missing component with specific features (commands, tokens, agent status). I should have asked earlier "what features did your dashboard have?" instead of assuming they meant the route.

## Lessons Learned

- Uncommitted files in /tmp are one reboot away from deletion — always commit or copy to persistent storage
- When user says "it's gone," check WSL /tmp deployment — it may have surviving source copies
- Windows PATH with parentheses breaks `bash -c` in WSL — use `-lc` or explicit PATH setting
